# 委派编排参考

## 难度分级

| 等级 | 典型特征 | 推荐 agent | 推荐模型 | reasoning_effort |
| --- | --- | --- | --- | --- |
| L1 | 单文件、小修复、上下文清晰、失败代价低 | `worker` | `gpt-5.4-mini` | `low` 或 `medium` |
| L2 | 常规功能或 bugfix，2 到 5 个文件，边界基本明确 | `worker` | `gpt-5.4` | `medium` 或 `high` |
| L3 | 跨模块改动、隐性耦合较多、验证成本高 | `worker` | `gpt-5.5` | `high` |
| L4 | 架构调整、迁移、需求仍在收敛、失败代价高 | 主线程先收敛方案，再派 `worker` | `gpt-5.5` | `xhigh` |

分级时至少看四个维度：

- 改动范围
- 代码陌生度
- 失败代价
- 验证成本

## Explorer 与 Worker 的分工

- `explorer`：用于回答具体代码库问题，例如“配置从哪里进来”“这个接口被哪些模块调用”“测试入口在哪里”。
- `worker`：用于实现、修测试、补脚本、调整配置和完成明确 write set 内的修改。
- 如果需要先摸清代码路径，再做实现，先派 `explorer`，不要让 `worker` 在模糊范围里边找边改。

## Worker Prompt 清单

至少写清楚以下字段：

- 目标
- 非目标
- 负责文件或模块
- 禁止改动范围
- 验证要求
- 最终回报格式

固定提醒：

- 你不是唯一在代码库里的 agent，不要回滚他人改动。
- 如遇到 ownership 外的阻塞，只报告，不擅自扩大改动范围。

## 回收检查单

- 返回的修改是否落在约定 ownership 内。
- 验证是否真的运行，而不是只口头声称。
- 是否引入跨 worker 的冲突或覆盖他人改动风险。
- 最终回答是否把结果、风险、未完成项分开说明。
