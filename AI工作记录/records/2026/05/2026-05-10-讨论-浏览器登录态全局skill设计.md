## 任务输入摘要

- 最终结果：头脑风暴一个“统一管理浏览器登录态”的全局 skill 设计方向，重点考虑浏览器爬虫脚本的复用、隔离和维护成本。
- 现有素材：项目已有的 skill 结构、工作记录模板、`skill-backlog.md`，以及用户最近频繁编写浏览器爬虫脚本这一背景。
- 明确约束：当前以方案讨论为主，不直接创建新 skill；登录态不应混入项目仓库；需要便于多个脚本复用和统一维护。
- 完成标准：形成几种可选方案，明确推荐架构、存储位置、数据模型、刷新机制和风险边界。
- 产出后动作：若方案成立，再决定是否正式创建全局 skill，并补充脚本、引用文档和校验流程。

## 背景

用户最近做了很多浏览器爬虫脚本，希望有一个统一的地方存储登录态，避免每个脚本重复登录、重复维护 cookie 或 profile。

## 关键过程

- 读取了 `skill-creator`，确认如果后续做成 skill，应保持 `SKILL.md` 精简，把脚本、引用文档和运行态数据分层管理。
- 读取了项目记录规则，确认本次讨论需要记录到项目内 `AI工作记录/records/YYYY/MM/`，并在出现稳定模式时同步更新 `skill-backlog.md`。
- 识别出这类能力的关键设计面不是单点存储，而是四件事一起定义：
  - 登录态存储格式：`storageState`、浏览器 profile，或两者并存。
  - 会话命名与隔离：按站点、环境、账号、浏览器维度区分。
  - 可用性校验：启动前如何判断会话是否仍有效。
  - 刷新与清理：失效后如何重新登录、更新、删除。

## 当前判断

- 更稳妥的方向是做成“全局 skill + 全局数据目录”的组合，而不是把登录态直接放在某个 skill 文件夹或项目仓库里。
- 推荐优先支持 Playwright 的 `storageState`，因为体积小、可移植、脚本接入成本低；对难缠站点再补充 profile 模式作为 fallback。
- 运行态数据应与 skill 定义分离，避免将真实登录态随 skill 同步、导出或提交。

## 2026-05-10 追加：任务复盘

### 这次真正的目标

- 不是单纯“保存一次 HTOJ 登录态”，而是把“浏览器登录态的统一管理”沉淀成可跨项目复用的全局能力，并把首次建档、后续刷新、实际站点验证都串成一条尽量顺手的路径。

### 哪些信息一开始就该给

- 如果一开始就明确“最终希望是一条命令，第一次和后续不要区分”，可以少走一轮 `-CreateIfMissing` 的过渡设计。
- 如果一开始就给出“用哪个真实业务页面判断是否已登录”，例如题目详情页，而不是首页，登录态验证逻辑会更早收敛。
- 如果一开始就说明主要使用 Playwright 和 Windows PowerShell 5/7 兼容要求，可以更早避免脚本调用和编码细节上的返工。

### 哪些返工来自需求或验收标准不完整

- `refresh_login.ps1` 第一版把“首次建档”和“后续刷新”分开，说明当时对最终 UX 的验收标准还不够明确。
- 登录态验证先在首页上判断，后面又切到题目详情页，说明“哪个页面最能代表真实登录状态”这个验收条件是后补的。
- PowerShell 脚本里关于参数透传、编码和子脚本调用的几次修正，反映出运行环境约束虽然已知，但没有在开始实现前显式写成检查清单。

### 下次怎么做会更好

- 在设计此类工具时，先把“用户最终只想执行哪一条命令”写成明确验收标准，再决定脚本参数结构。
- 对需要登录态验证的站点，优先选一个高信号页面作为 `checkUrl`，不要默认拿首页当验证页。
- 对 PowerShell + Playwright 组合，后续可以固定一份实现约束：
  - 子脚本调用优先用 splatting，不用字符串参数数组伪装命名参数；
  - 非 ASCII 提示文案要考虑 PowerShell 5 编码表现；
  - 先做 `DryRun` 路径，再做真实交互路径。

### 是否值得继续模板化或 skill 化

- 值得，且这次已经完成第一步：`browser-session-manager` 已经从想法变成了可安装的全局 skill。
- 下一步最值得补的是两个稳定能力：
  - 一个 `verify_session` 脚本，用 `checkUrl` + `checkSelector` 统一判断登录态；
  - 一份站点接入模板，让新增站点时只需要填 `site`、`checkUrl`、`checkSelector` 和少量备注。
