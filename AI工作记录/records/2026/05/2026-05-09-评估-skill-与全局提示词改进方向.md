## 背景

用户提供了一组关于 AI 协作效率的判断，核心观点是效率瓶颈常常不在模型能力，而在任务下达粒度过小、经验沉淀过早 skill 化。用户希望基于这些信息，判断当前 `record-and-reflect-review` skill 以及全局 `AGENTS.md` 提示词后续应该往哪个方向优化。

## 关键过程

- 阅读了当前仓库中的 `SKILL.md`、`references/global-agents-rules.md`、`references/record-template.md`、`skill-backlog.md`，以及两条历史记录。
- 对照现有内容后判断：当前 skill 已经较强地覆盖了“记录摘要、回顾历史、沉淀 skill 候选、维护全局规则”这条链路，但对“任务开始时如何把需求组织成可交付单元”覆盖不足。
- 进一步判断这次更适合先增强模板和触发条件，而不是立即拆出一个全新的 skill。
- 发现一个结构不一致问题：仓库根目录存在 `skill-backlog.md`，但 `SKILL.md` 与全局规则文本多处写的是 `AI工作记录/skill-backlog.md`，后续需要统一。

## 结果

- 形成的主要判断：
  - `record-and-reflect-review` 下一步最值得补的是“任务输入模板”和“固定复盘模板”，而不是继续扩写泛化规则。
  - 全局 `AGENTS.md` 不宜写成“任何任务都先提问”，而应更明确鼓励用户和 AI 尽量以“目标、输入、约束、验收、输出/后续动作”五要素组织任务。
  - “什么时候升级成 skill”的判断标准已经隐含存在，但可以写得更显式，避免过早 skill 化。
- 本次没有直接修改 skill 或全局规则正文；先记录判断，供后续确认后实施。

## 追加判断

- 用户进一步明确了更实用的职责拆分方向：
  - 全局提示词负责尽量让每次任务输入完整，减少来回澄清。
  - `record-and-reflect-review` 更适合聚焦在过程记录，而不是同时承担完整复盘方法论。
  - 复盘与改进更适合在任务完成后单独触发；是否做成独立 skill，可以等模板稳定后再决定。
- 基于这次补充，当前更推荐的实施顺序是：
  - 先增强全局 `AGENTS.md` 对“可交付任务输入”的引导。
  - 保持现有记录 skill 以记录为主。
  - 把复盘先做成固定模板或流程，等高频复用后再考虑独立 skill。

## 可复用经验

- 协作优化的第一优先级通常不是新增 skill，而是先把任务入口和任务收尾模板化。
- 记录类 skill 如果只覆盖“事后总结”，容易错过真正影响效率的“事前任务定义”环节。
- 当规则文本与仓库真实结构不一致时，先记录为结构债务，再决定是否迁移文件或修正文案。

## Skill 观察

- 是否出现新 skill 候选：否，更像是现有 `record-and-reflect-review` 的增强项。
- 是否应该优化已有 skill：是，建议把现有 skill 的重点收敛到记录；任务输入约束更多放到全局提示词；复盘能力先模板化，成熟后再决定是否拆成单独 skill。
- 触发条件或典型用户说法：根据一次对话总结当前 skill 和全局提示词的改进方向；希望减少来回澄清，把任务直接说成可交付单元。
