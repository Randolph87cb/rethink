## 背景

用户希望把 Codex 的主线程收敛为“讨论方向和规则”的控制平面，把具体代码修改、验证等执行工作交给 subagent，并且不是固定一个模型，而是按任务难度动态选择模型和 reasoning effort。

## 关键过程

- 读取了 `record-and-reflect-review` 的说明、记录模板、项目 `skill-backlog.md`，确认本项目默认需要按线程维护摘要记录。
- 阅读了 2026-05-09 的历史记录，发现此前已经在讨论“任务入口、规则分层、skill 触发条件”一类问题，这次是在此基础上继续往“多 agent 编排模式”推进。
- 识别到一个关键硬边界：当前 Codex 的上层工具约束要求，只有用户明确提出 subagent / delegation / parallel work 时，主线程才应调用 `spawn_agent`。因此想把“默认所有代码任务都自动委派”做成无条件行为，单靠项目级 AGENTS 或 skill 可能不稳，更适合设计成显式触发的工作模式。

## 结果

- 初步判断：这件事最适合拆成三层来做：
  - 项目级 AGENTS：定义“委派模式”的触发语、主线程职责边界、例外情况。
  - 本地 skill：沉淀任务分解模板、难度分级标准、worker prompt 模板、集成与回收检查单。
  - 实际执行：当用户显式开启委派模式时，由主线程负责任务拆解和模型选择，subagent 负责实现、验证、汇报。
- 进一步判断：如果希望“像工作记录 skill 一样默认会用”，不能只安装 skill 本身，而要同时具备三层：
  - 全局安装该 skill，使其在所有项目中可用；
  - 在 skill 的 `agents/openai.yaml` 中开启隐式触发；
  - 在全局 `AGENTS.md` 与项目 `AGENTS.md` 中明确写出默认启用条件和主线程职责边界。
- 进一步判断：即使做成全局 skill，也只能把“默认工作模式”约束得很强，不能绕过上层工具对 `spawn_agent` 的使用边界；因此更稳妥的方案是把它设计成显式委派模式，而不是声称所有代码任务都会无条件自动分发给 subagent。
- 本次未修改代码或 skill，仅形成方案草图。

## 可复用经验

- “主线程只做编排、执行全交给 worker”本质上不是单条提示词，而是一套 operating mode；更适合设计成显式模式，而不是隐式默认。
- 如果要按难度选模型，先定义统一分级维度比直接拍脑袋选模型更重要。建议至少考虑：改动范围、代码陌生度、失败代价、验证成本。
- 多 agent 想稳定，关键不在多开几个 worker，而在于写清楚 ownership、文件边界、验收条件和等待时机。

## Skill 观察

- 是否出现新 skill 候选：是，候选方向是“subagent delegation orchestrator”或“delegation mode”类全局 skill。
- 是否应该优化已有 skill：暂不直接改 `record-and-reflect-review`；它更适合记录和沉淀这类模式，不适合本身承担委派编排。
- 触发条件或典型用户说法：主线程只讨论方向和规则；具体修改交给 subagent；根据任务难度定模型和思考深度。
